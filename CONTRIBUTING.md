# Contributing to DriveBy Studios

Thank you for your interest in contributing to the DriveBy Studios ecosystem. We are committed to building high-performance, architecture-driven software. To maintain the quality of our projects, please adhere to the following guidelines.

## 🚀 Our Philosophy
We believe in "Drive-by Development": **Identify, Architect, Execute.** Our code must be performant, readable, and engineered for high-intensity environments. Whether you are working on *CiPhyxs* or our networking frameworks, please prioritize efficiency and clean system design.

## 🛠️ Getting Started
1.  **Fork the Repository:** Create a personal fork of the project.
2.  **Create a Branch:** Use a descriptive branch name (e.g., `feature/sat-narrowphase` or `fix/vulkan-sync`).
3.  **Adhere to Standards:** * **C++20 Compliance:** Ensure all code uses modern C++20 features where appropriate.
    * **Performance First:** Avoid unnecessary allocations. If your change affects a hot path, include performance metrics or benchmarks if possible.
    * **Clean Architecture:** Keep logic modular and well-documented.

## 📝 Pull Request Process
* **Check CI:** Ensure your branch passes all automated checks in our CI pipeline.
* **Documentation:** If you add a new feature, update the relevant documentation or headers.
* **Review:** Open a Pull Request (PR) against the `main` or `develop` branch. Our team will review the code specifically for performance, API consistency, and architectural impact.

## 🐛 Reporting Bugs
If you encounter an issue, please use the provided **Bug Report** template. Include:
* **Minimal Reproduction:** A small example or steps that trigger the issue.
* **Environment Details:** OS, Compiler, GPU/Driver versions (critical for Vulkan/graphics issues).
* **Expected vs. Actual:** Clear details on what should happen versus what is occurring.

## 🤝 Community & Support
Questions? Reach out via the [DriveBy Studios GitHub Discussions](https://github.com/orgs/DriveBy-Studios/discussions). We value collaborative engineering and welcome thoughtful feedback on our architecture.
